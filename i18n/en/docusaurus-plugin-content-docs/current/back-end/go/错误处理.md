# Error Handling - The Go Way

Go doesn't have exceptions, only errors. This seemingly simple approach leads to clearer, more reliable code.

## Target Errors as Values

In Go, an error is a value:

```go
type error interface {
    Error() string
}

// Function returns error
func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}

// Handle error
result, err := divide(10, 0)
if err != nil {
    fmt.Println("Error:", err)
    return
}
fmt.Println("Result:", result)
```

## Wrench Creating Errors

### 1. errors.New

```go
err := errors.New("something went wrong")
```

### 2. fmt.Errorf (formatted)

```go
name := "John"
err := fmt.Errorf("user %s not found", name)
```

### 3. Custom Error Types

```go
type NotFoundError struct {
    Resource string
    ID       int
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("%s with ID %d not found", e.Resource, e.ID)
}

// Usage
err := &NotFoundError{Resource: "user", ID: 123}
```

## Checkmark Error Handling Patterns

### Pattern 1: Return Early

```go
func doSomething() error {
    result, err := step1()
    if err != nil {
        return err  // Return early
    }
    
    result2, err := step2(result)
    if err != nil {
        return err  // Return early
    }
    
    return step3(result2)
}
```

### Pattern 2: Wrap Errors

```go
import "errors"

func process() error {
    if err := doWork(); err != nil {
        return fmt.Errorf("processing failed: %w", err)
    }
    return nil
}

// Check wrapped error
if errors.Is(err, ErrNotFound) {
    // Handle not found
}
```

### Pattern 3: Sentinel Errors

```go
var (
    ErrNotFound   = errors.New("not found")
    ErrInvalid    = errors.New("invalid input")
    ErrPermission = errors.New("permission denied")
)

func findUser(id int) (*User, error) {
    user := db.Get(id)
    if user == nil {
        return nil, ErrNotFound
    }
    return user, nil
}

// Check specific error
if errors.Is(err, ErrNotFound) {
    // Handle not found specifically
}
```

## Warning Common Mistakes

### Mistake 1: Ignoring Errors

```go
// Bad: ignoring error
file, _ := os.Open("file.txt")  // Don't do this!

// Good: handle error
file, err := os.Open("file.txt")
if err != nil {
    log.Fatal(err)
}
defer file.Close()
```

### Mistake 2: Over-wrapping

```go
// Bad: too many layers
if err != nil {
    return fmt.Errorf("layer1: %w", fmt.Errorf("layer2: %w", err))
}

// Good: wrap once at boundary
if err != nil {
    return fmt.Errorf("database operation failed: %w", err)
}
```

### Mistake 3: Using panic

```go
// Bad: don't use panic for normal errors
if err != nil {
    panic(err)  // Don't!
}

// Good: return error
if err != nil {
    return err
}
```

## Lightbulb Best Practices

1. **Always check errors** - Never ignore them
2. **Return early** - Reduce nesting
3. **Wrap at boundaries** - Add context when crossing layer boundaries
4. **Use sentinel errors** - For common error types
5. **Custom error types** - When you need extra information
6. **Don't panic** - Use errors for normal control flow

## Whitecheckmark Summary

- Error handling is explicit in Go
- Errors are values, not exceptions
- Check errors immediately after function calls
- Wrap errors to add context
- Use errors.Is() and errors.As() for error inspection
