# Error Handling - The Go Way

Go doesn't have exceptions, only errors. This seemingly simple approach leads to clearer, more reliable code.

## 🎯 Errors as Values

In Go, an error is a value:

```go
type error interface {
    Error() string
}

// Function returns error
func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}

// Handle error
result, err := divide(10, 0)
if err != nil {
    fmt.Println("Error:", err)
    return
}
fmt.Println("Result:", result)
```

## 🔧 Creating Errors

### 1. errors.New

```go
err := errors.New("something went wrong")
```

### 2. fmt.Errorf (Formatting)

```go
name := "John"
err := fmt.Errorf("user %s not found", name)
```

### 3. Custom Error Types

```go
type ValidationError struct {
    Field   string
    Message string
}

func (e ValidationError) Error() string {
    return fmt.Sprintf("%s: %s", e.Field, e.Message)
}

// Usage
err := ValidationError{Field: "email", Message: "invalid format"}
```

## 📋 Error Handling Patterns

### 1. Check Error Immediately

```go
file, err := os.Open("data.txt")
if err != nil {
    log.Fatal(err)
}
defer file.Close()
```

### 2. Error Propagation

```go
func processFile(filename string) error {
    file, err := os.Open(filename)
    if err != nil {
        return fmt.Errorf("opening file: %w", err)
    }
    defer file.Close()
    
    // Process...
    return nil
}
```

### 3. Sentinel Errors

```go
var ErrNotFound = errors.New("not found")

func findUser(id int) (*User, error) {
    user := db.Get(id)
    if user == nil {
        return nil, ErrNotFound
    }
    return user, nil
}

// Check specific error
user, err := findUser(123)
if err == ErrNotFound {
    // Handle not found
}
```

## 🆚 panic and recover

Rarely used, for truly exceptional situations:

```go
func riskyOperation() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered from:", r)
        }
    }()
    
    // Something that might panic
    panic("something went wrong!")
}
```

**Rule:** Don't use panic for normal error handling!

## 💡 Best Practices

### 1. Return Errors, Not Print Them

```go
// ✅ Good: Let caller decide
func readConfig(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, err
    }
    // Parse...
}

// ❌ Bad: Hidden side effect
func readConfig(path string) *Config {
    data, err := os.ReadFile(path)
    if err != nil {
        log.Println(err)  // Hidden!
        return nil
    }
    // ...
}
```

### 2. Wrap Errors with Context

```go
// Add context when propagating
if err != nil {
    return fmt.Errorf("processing order %d: %w", orderID, err)
}
```

### 3. Check Errors in Tests

```go
func TestDivide(t *testing.T) {
    _, err := divide(10, 0)
    if err == nil {
        t.Error("expected error")
    }
}
```

## 📊 Comparison

| Approach | Language | Pros | Cons |
|----------|----------|------|------|
| Exceptions | Java/Python | Automatic propagation | Hidden control flow |
| Error values | Go | Explicit, clear | Verbose |
| Result types | Rust | Type-safe | Complex |

**Go philosophy:** Explicit is better than implicit.
